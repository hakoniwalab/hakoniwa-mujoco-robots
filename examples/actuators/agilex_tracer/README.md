# AgileX Tracer Rover Samples

These samples validate an AgileX Tracer V1 rover model generated by the sibling
`hakoniwa-mbody-registry` checkout.

The intended flow is staged:

1. Validate the generated MJCF, collision/contact setup, and wheel actuators in
   MuJoCo only.
2. Run a Hakoniwa asset that receives `geometry_msgs/Twist` on `cmd_vel`.
3. Send commands from Python and confirm the rover moves after `hako-cmd start`.

The sample keeps the robot MJCF generated. Tracer-specific additions such as
actuators, primitive collision geoms, contact excludes, and the minimal world are
tool-chain inputs in `hakoniwa-mbody-registry`, not a hand-maintained MJCF fork.

## What Is Shared

The left/right wheel actuator bindings use the existing common
`JointActuatorImpl` JSON mechanism:

```text
config/actuator/joint/agilex_tracer_left_wheel.json
config/actuator/joint/agilex_tracer_right_wheel.json
```

The Twist-to-wheel conversion uses the shared rover helper:

```text
include/robots/rover/differential_drive_kinematics.hpp
```

Tracer currently sets `left_wheel_sign=-1` and `right_wheel_sign=1` because the
left wheel joint orientation is opposite to the right wheel in the converted
model. The helper is generic, but the Tracer dimensions and signs are still
specified in the sample code. Sensor output is not part of this sample yet; add
joint state, odometry, IMU, or LiDAR only after the command path is stable.

## Prerequisite

Generate the Tracer model in the sibling `hakoniwa-mbody-registry` checkout:

```bash
cd ../hakoniwa-mbody-registry
PATH=$PWD/.venv/bin:$PATH HAKO_SKIP_FETCH=1 ./tools/forge.sh sources/agilex_tracer.yaml bodies/agilex_tracer/generated
```

This example expects the minimal world generated from:

```text
../hakoniwa-mbody-registry/bodies/agilex_tracer/config/actuators.yaml
../hakoniwa-mbody-registry/bodies/agilex_tracer/config/collision_primitives.yaml
../hakoniwa-mbody-registry/bodies/agilex_tracer/config/contact_excludes.yaml
../hakoniwa-mbody-registry/bodies/agilex_tracer/config/mujoco_world.yaml
```

Expected generated model:

```text
../hakoniwa-mbody-registry/bodies/agilex_tracer/generated/tracer_v1.minimal_world.xml
```

## Build

From the `hakoniwa-mujoco-robots` repository root:

```bash
cmake --build src/cmake-build --target agilex-tracer-rover-example
cmake --build src/cmake-build --target rover-twist-hakoniwa-asset
```

## Quick Check

Run this first. It does not open the viewer or require Hakoniwa shared memory:

```bash
./src/cmake-build/examples/actuators/agilex_tracer/agilex-tracer-rover-example --check
```

Expected result:

```text
time=  1.000 base=( 0.055,  0.000,  0.142) cmd=(0.200 m/s, 0.000 rad/s) wheel_target=(-2.439,  2.439) ...
check ok
```

The exact wheel velocity can vary slightly. The important signs are:

- the model and both wheel actuator configs load;
- `base.x` is positive after the forward command;
- the process prints `check ok`.

```bash
  config/actuator/joint/agilex_tracer_right_wheel.json
```

## Controls

```text
w / s  : increase / decrease linear.x target
a / d  : increase / decrease angular.z target
Space  : stop linear and angular targets
r      : reset simulation state and targets
p      : pause / resume physics
h      : show help
q / Esc: quit
```

The sample converts the two targets to wheel angular velocities with:

```text
left_target  = -(linear.x - angular.z * wheel_separation / 2) / wheel_radius
right_target =  (linear.x + angular.z * wheel_separation / 2) / wheel_radius
```

The actual code uses `hako::robots::rover::ToWheelVelocityTargets()` so the same
kinematics are shared with the Hakoniwa Twist asset.

## Hakoniwa Twist Asset

The Hakoniwa sample opens a MuJoCo viewer and receives `geometry_msgs/Twist`
directly:

```bash
cmake --build src/cmake-build --target rover-twist-hakoniwa-asset
```

Use three terminals. Start `hako-cmd start` only after the asset and sender have
both registered and are waiting.

Terminal 1, plant asset and Conductor owner:

```bash
./src/cmake-build/examples/actuators/agilex_tracer/rover-twist-hakoniwa-asset
```

This opens the MuJoCo viewer immediately. Simulation time does not advance until
`hako-cmd start`; before start, the window shows the loaded Tracer model waiting
at the initial pose.

Expected signs:

```text
hako_conductor thread start
hako_asset_register :RoverTwistAsset
asset(RoverTwistAsset) is registered.
[INFO] Starting rover twist asset with:
  viewer       : enabled
[INFO] asset=RoverTwistAsset cmd_vel=cmd_vel
WAIT START
```

Terminal 2, Python command sender:

```bash
python3 examples/actuators/agilex_tracer/send_rover_twist.py --linear-x 0.2 --duration-sec 3
```

Expected signs:

```text
hako_asset_register :RoverTwistSender
asset(RoverTwistSender) is registered.
[INFO] Rover Twist sender is registered.
WAIT START
```

Terminal 3, start simulation time:

```bash
/usr/local/hakoniwa/bin/hako-cmd start
```

The command channel is:

```text
RoverTwistAsset/cmd_vel  geometry_msgs/Twist
```

Expected signs after start:

```text
PDU DATA CREATED
[INFO] Rover Twist Hakoniwa asset started.
time=  0.002 base=(0.000, 0.000, 0.142) cmd=(0.000, 0.000) wheel=(-0.000, 0.000)
time=  3.002 base=(0.300, 0.000, 0.142) cmd=(0.200, 0.000) wheel=(-2.439, 2.439)
```

The sender stops after its duration and sends a zero command. The plant asset is
long-running; end it with `Ctrl-C` in terminal 1 when you are done. Do not use
`hako-cmd stop` as process cleanup unless your local runtime flow explicitly
supports it.

For headless runtime checks, pass `--no-viewer` or set
`HAKO_ROVER_TWIST_ENABLE_VIEWER=0`:

```bash
./src/cmake-build/examples/actuators/agilex_tracer/rover-twist-hakoniwa-asset --no-viewer
```

## Troubleshooting

- If the Hakoniwa commands fail with shared-memory errors, run the asset, sender,
  and `hako-cmd start` in a normal terminal outside restricted sandboxes.
- If the viewer does not open, use `--no-viewer` to separate Hakoniwa/PDU
  runtime issues from local GUI/OpenGL issues.
- If `hako-cmd start` is issued before both assets show `WAIT START`, restart the
  runtime and follow the order above.
- If `--check` fails before Hakoniwa is involved, inspect the generated
  `tracer_v1.minimal_world.xml`, actuator configs, collision primitives, and
  contact excludes first.
- If the rover moves backward for a positive `linear.x`, check the wheel signs in
  the `DifferentialDriveKinematics` settings.

## Current Scope

This sample intentionally keeps the generated robot MJCF as tool output. The
world file is produced by `hakoniwa-mbody-registry` from the original Tracer
Xacro/URDF plus:

- `bodies/agilex_tracer/config/actuators.yaml`
- `bodies/agilex_tracer/config/collision_primitives.yaml`
- `bodies/agilex_tracer/config/contact_excludes.yaml`
- `bodies/agilex_tracer/config/mujoco_world.yaml`

Do not hand-maintain a Tracer-specific fork of the generated MJCF for normal
Recipe flow.
